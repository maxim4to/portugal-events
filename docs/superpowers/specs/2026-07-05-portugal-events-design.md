# Спек: «Куда поехать в Португалии» — каталог мест и афиша событий

Дата: 2026-07-05
Статус: утверждён

## Цель

Личный сайт для планирования поездок по Португалии из Лиссабона на машине. Два типа контента:

- **Статические места** — пляжи, хайкинг, замки, озёра, смотровые площадки, городки. Заносятся единоразово, дальше пополняются точечно.
- **Динамические события** — концерты, фестивали, выставки, ярмарки. Собираются автоматически из источников (телеграм-каналы, сайты афиш, RSS) с ручной модерацией.

Главный сценарий использования: открыть сайт и решить, чем заняться в ближайшие выходные.

## Ключевые решения

| Вопрос | Решение |
|---|---|
| Аудитория | Только владелец (и близкие). Без аккаунтов, регистрации и мультиюзерности. |
| Архитектура | Статический сайт + данные как JSON-файлы в git. Без сервера и БД. |
| Хостинг | GitHub Pages (репозиторий публичный — данные видны всем, для этого контента приемлемо). |
| Наполнение мест | AI-генерация (Claude с веб-поиском) + ручной отбор владельцем. |
| Сбор событий | Локальный агент Claude Code по расписанию + модерация через pull request. Без Claude API и ключей. |
| Стек | Astro (сайт), TypeScript + zod (валидация), Leaflet + OpenStreetMap (карта), Open-Meteo (погода). |

## Архитектура

Один репозиторий, «бэкенд» — это git + GitHub Actions:

```
portugal-events/
├── data/                    # «база данных» — JSON-файлы
│   ├── places/              # места, один файл на регион (lisboa.json, alentejo.json, ...)
│   ├── events/              # события, один файл на месяц (2026-07.json, ...)
│   │   └── archive/         # прошедшие события
│   ├── sources.json         # реестр источников событий
│   └── geo/cities.json      # справочник городов Португалии с координатами
├── site/                    # статический сайт (Astro)
├── pipeline/                # детерминированные проверки: validate.ts, dedupe-check.ts
├── .claude/skills/
│   ├── ingest-events/       # инструкция агенту: сбор событий
│   └── generate-places/     # инструкция агенту: генерация базы мест
├── docs/superpowers/specs/  # этот документ и будущие спеки
└── .github/workflows/
    └── deploy.yml           # единственный workflow: сборка Astro → GitHub Pages при пуше в main
```

Поток данных: источники → локальный агент Claude Code (по расписанию) → pull request → ревью владельцем → merge в main → GitHub Actions пересобирает сайт → GitHub Pages.

## Модель данных

Все схемы описаны в zod (единый пакет, используется и проверками pipeline, и сборкой сайта). Кривой JSON не проходит ни валидацию перед коммитом, ни сборку.

### Place (место)

- `id` — стабильный слаг (`praia-da-ursa`)
- `name` — название
- `type` — `beach | hike | castle | lake | viewpoint | town | nature | other`
- `region` — регион Португалии
- `coords` — широта/долгота
- `description` — 2–3 абзаца: зачем ехать, что там делать
- `driveMinutesFromLisbon` — время в пути на машине от Лиссабона
- `visitDurationHours` — сколько времени закладывать на визит
- `bestSeason` — лучшие месяцы/сезон
- `tags` — свободные теги (`с детьми`, `закат`, `бесплатно`, ...)
- `links` — официальный сайт, AllTrails и т.п.
- `status` — `candidate` (сгенерировано AI, ждёт отбора) | `approved` (показывается на сайте)

### Event (событие)

- `id` — слаг
- `title` — название
- `type` — `concert | festival | exhibition | market | sport | other`
- `dateStart` / `dateEnd` — ISO-даты
- `city` + `venue` — город (матчится со справочником `cities.json` для координат) и площадка
- `price` — цена/бесплатно, свободный текст
- `url` — ссылка на источник или билеты
- `description` — 1–2 предложения
- `sourceId` — откуда пришло
- `dedupeHash` — хэш от нормализованных название+дата+город

### Source (источник)

- `id`, `name` — идентификатор и человекочитаемое название
- `type` — `telegram | website | rss`
- `url` — для телеграма публичный канал, читается через `t.me/s/<канал>`
- `hint` — подсказка агенту (что на странице искать, какой раздел)
- `enabled` — вкл/выкл

Добавить источник = добавить объект в `sources.json`.

## Наполнение базы мест (единоразово, потом точечно)

Процесс, а не код. Интерактивная сессия Claude Code со скиллом `generate-places`: агент с веб-поиском проходит по матрице «категория × регион» (пляжи Алентежу, замки Центра, хайки Синтры, ...) и генерирует JSON по схеме — ориентир 150–300 мест на страну, каждое со статусом `candidate`. Владелец просматривает пачками, выкидывает мусор, правит описания, переводит оставшееся в `approved`. Пополнение базы позже — тем же скиллом точечно.

## Пайплайн событий

1. **Запуск по расписанию.** Headless-сессия Claude Code в директории проекта: `claude -p "/ingest-events"` через launchd/cron или встроенные scheduled tasks приложения Claude Code. Ориентир — раз в день утром.
2. **Инструкция вместо кода.** Логика сбора — в скилле `ingest-events`: прочитать `sources.json`, обойти включённые источники (телеграм через `t.me/s/`, сайты через WebFetch, RSS как RSS), извлечь события по схеме, нормализовать даты и города, отбросить прошедшие и дубли (по `dedupeHash` внутри пачки и против существующих `data/events/`), записать в файлы месяцев. Агент может переходить по ссылкам из постов, чтобы уточнить дату/цену.
3. **Детерминированные проверки.** Перед коммитом агент обязан прогнать `pipeline/validate.ts` (zod-схемы) и `pipeline/dedupe-check.ts` (уникальность хэшей). Страховка от галлюцинаций: невалидные данные не проходят.
4. **Модерация.** Агент коммитит в ветку и открывает PR через `gh` с читаемым списком новых событий. Владелец ревьюит (в том числе с телефона), merge запускает пересборку сайта. Переключение на «коммит сразу в main» — одна строчка в скилле.
5. **Уборка.** Та же инструкция: завершившиеся события перемещаются в `data/events/archive/`.

Ограничения (осознанные):

- Сбор идёт только когда Mac включён и не спит. Пропущенный день не критичен — события живут в источниках неделями. При желании скилл переезжает в облачные routines без переделки системы.
- `t.me/s/` работает только для публичных каналов и отдаёт последние ~20 постов — при ежедневном запуске достаточно. Приватные каналы — вне скоупа v1.
- Если источник не отвечает несколько запусков подряд, агент фиксирует это в PR-описании/issue, чтобы сбой не потерялся молча.

## Фронтенд

Astro: статический HTML + точечные островки лёгкого JS. При сборке все данные склеиваются в один JSON-бандл (сотни килобайт — клиентские фильтры и поиск работают мгновенно, бэкенд не нужен).

Страницы:

- **«Выходные» (главная)** — ближайшие сб-вс: события на эти даты + подборка мест с учётом сезона и погоды. Прогноз тянется на клиенте из Open-Meteo (бесплатно, без ключа): дождь → музеи и выставки вместо пляжей.
- **Каталог мест** — карта (Leaflet + OSM) и список; фильтры: тип, регион, время в пути от Лиссабона, теги, сезон.
- **Афиша** — список/календарь событий; фильтры: тип, город, даты.
- **Карточка места/события** — описание, мини-карта, ссылки, кнопка «маршрут в Google Maps».
- **Личное** — избранное, «был здесь», «скрыть»: localStorage + экспорт/импорт JSON для переноса между устройствами.

## Надёжность и тесты

- zod-схемы — единственный источник правды о формате; валидация в pipeline-проверках и при сборке сайта.
- Юнит-тесты pipeline: нормализация дат, дедуп, матчинг городов — на фикстурах реальных HTML источников.
- Сборка сайта в CI — смоук-тест: битые данные валят деплой до публикации.

## Этапы реализации

Каждый этап заканчивается работающим сайтом.

1. **Скелет** — репозиторий, zod-схемы, Astro-сайт с десятком мест руками, деплой на GitHub Pages.
2. **База мест** — скилл `generate-places`, AI-генерация по регионам, отбор, каталог с картой и фильтрами.
3. **Пайплайн событий** — `sources.json` с 3–5 источниками, скилл `ingest-events`, проверочные скрипты, PR-флоу, настройка расписания, страница афиши.
4. **«Выходные»** — главная с погодой, избранное в localStorage.
5. **Полировка** — архивация, сигналы о сбоях источников, мобильная вёрстка.

## Вне скоупа v1

- Аккаунты, синхронизация избранного через сервер.
- Приватные телеграм-каналы (нужен Telegram API).
- Уведомления/дайджесты (например, письмо «что в эти выходные») — возможное развитие.
- Страны кроме Португалии.
