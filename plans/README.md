## Findings reported but not planned

(Real findings; lower leverage. Plan on request.)

- **Astro 5 → 7 major-version lag** (`npm audit`: 5 advisories incl. XSS in
  `define:vars`/slots/spread-props, fixed only in 7.x). None of the vulnerable features
  are used by this static site, so exploitation is not currently possible; upgrade is
  defensive hygiene. Effort M (breaking-change migration + full page retest).
- **Client-script logic untested** — `PhotoGallery.astro` (~190 lines of swipe/wheel
  state machine) and `PlacesExplorer.astro` (~190 lines of filter/map wiring) have no
  tests; only the extracted `matchesFilter` predicate is covered. The fix worth doing
  is extraction-then-test (pure helpers into `site/lib/`), effort M each.
- **No README.md** — the repo has only the agent-facing CLAUDE.md; a short
  human-facing README (what/run/architecture) would help since the repo is public.
  Effort S.
- **No lint/format tooling** — no ESLint/Prettier/pre-commit. Cheap to add, but a
  single-maintainer repo with consistent style loses little without it. Effort S.
- **Firebase RTDB rules verification** (operational, not code): project notes say
  scoped rules (`spaces/$space` read+write only) may still not be applied in the
  Firebase console — the DB may be in wide-open test mode. Cannot be checked from the
  repo. **The maintainer should verify this in the console**; plan 004 adds the
  client-side hardening half.

## Direction options (maintainer's call, not ranked against fixes)

- **Weekend weather integration** — the design doc explicitly promises Open-Meteo
  forecasts ("дождь → музеи и выставки вместо пляжей", spec line ~116) and it's stage 4
  of the staged plan; nothing in code yet. Natural MVP: build- or client-time Lisbon
  forecast + a rainy-day collection banner on the home page. Effort L (coarse).
- **Sync favorites into the shared space** — place detail pages already have a working
  "В избранное" button (`site/pages/places/[id].astro:65`, localStorage-only), while
  "visited" syncs via Firebase. Favorites don't survive device changes; reusing the
  `spaces/<spaceId>/favorites/` path mirrors an existing, proven pattern. Effort M.
- **Filter state in the URL** — `PlacesExplorer` already *reads* `?type=`/`?tag=` on
  load but never writes state back, so filtered views can't be bookmarked/shared.
  Completing the round-trip via `history.replaceState` is a small, self-contained UX
  win. Effort S–M.

## Findings considered and rejected

- "Event sources dormant / calendar empty" (subagent report): **wrong** — 8 enabled
  sources in `data/sources.json`, 41 real events Jul–Oct 2026 in `data/events/`.
- "Event `image` field half-built, ingestion doesn't populate it": **wrong** —
  `.claude/skills/ingest-events/SKILL.md:51-54` instructs og:image capture and most
  events have images.
- spaceId/localStorage race conditions in `visited.ts` (clearing storage mid-await):
  theoretical timing, silent-fail worst case, feature degrades gracefully — not worth
  code churn.
- localStorage quota handling for favorites/visited: arrays of short slugs; quota is
  unreachable in practice.
- Pipe-delimited `data-tags` corruption if a tag contains `|`: tags are a closed
  vocabulary (11 fixed Russian words, none with pipes).
- Committed Firebase web config: by design, standard for Firebase web apps.
- `astro.config.mjs` `site:` URL "stale": explicitly intentional per the maintainer —
  do not change.
- ADR for the Firebase RTDB choice: the design doc + CLAUDE.md already record the
  architecture; a personal repo gains little from ADR formalism.
- Inlined places JSON / data-* payload size: measured fine at 144 places; revisit only
  past ~500 places.
